## 1. What Is This Project?

This is the **backend API** for a finance dashboard. Think of it as the brain behind a web app where people can:

- Register and log in
- Record financial transactions (income and expenses)
- See analytics like "how much did I spend this month?" or "what category costs the most?"

It doesn't have a frontend. It's a pure REST API вҖ” it receives JSON requests and sends JSON responses. A frontend (React, Next.js, etc.) would call these endpoints.

There are **three types of users** (called "roles"):

| Role     | What they can do |
|----------|------------------|
| VIEWER   | Can look at transactions. That's it. Read-only access. |
| ANALYST  | Everything a VIEWER can do, PLUS they can see the dashboard analytics (charts, summaries, trends). |
| ADMIN    | Can do everything. Create/edit/delete transactions, manage users, see analytics. The boss. |

## 3. What Happens When the Server Starts

When you run `npm run dev`, this is the exact sequence:

### Step 1: `server.ts` runs

```typescript
import "dotenv/config";      // Loads .env file into process.env
import { app } from "./app.js";

const PORT = parseInt(process.env.PORT || "3000", 10);

app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});
```

The very first line `import "dotenv/config"` is critical. It reads your `.env` file and makes all the values available as `process.env.DATABASE_URL`, `process.env.JWT_SECRET`, etc. **This must happen before anything else** because other files (like `db.ts`) need these values.

### Step 2: `app.ts` initializes Express

```typescript
const app = express();

// Security and parsing middleware
app.use(helmet());          // Adds security headers (prevents common attacks)
app.use(cors());            // Allows requests from other domains (your frontend)
app.use(express.json());    // Parses JSON request bodies

// Health check (no auth needed)
app.get("/health", ...);

// Mount all route groups
app.use("/api/auth",         authRoutes);                                    // Public
app.use("/api/users",        verifyToken, requireRole("ADMIN"), userRoutes); // Admin only
app.use("/api/transactions", verifyToken, transactionRoutes);                // Logged in
app.use("/api/dashboard",    verifyToken, dashboardRoutes);                  // Logged in

// Error handler goes LAST
app.use(errorHandler);
```

Notice the pattern:
- `/api/auth` has **no middleware** before it вҖ” anyone can register/login
- `/api/users` has **two middleware** вҖ” you must be logged in AND be an ADMIN
- `/api/transactions` has **one middleware** вҖ” you just need to be logged in
- `/api/dashboard` has **one middleware** at app level, but the dashboard routes file adds another role check internally

The `errorHandler` is mounted **last** because it catches errors thrown by everything above it.

### Step 3: Nothing else happens until a request comes in

The server is now sitting there, listening on port 3000, waiting for HTTP requests.

## 5. Authentication вҖ” How Login and Tokens Work

### The Problem

HTTP is **stateless**. The server doesn't remember who you are between requests. So we need a way to say "I'm the user who logged in 5 minutes ago" on every request.

### The Solution: JWT (JSON Web Token)

When you log in, the server gives you a **token** вҖ” a long encoded string. You send this token with every future request. The server decodes it and knows who you are.

### The Login Flow

```
1. Client sends:    POST /api/auth/login
                    { "email": "alice@example.com", "password": "secret123" }

2. Server does:
   a. Find user by email in database
   b. Compare password with the stored hash (bcrypt.compare)
   c. If match вҶ’ create a JWT token containing { userId: "abc123", role: "VIEWER" }
   d. Send the token back

3. Client receives: { "token": "eyJhbGciOiJIUzI1NiIs..." }

4. Client stores the token (usually in localStorage or a cookie)

5. On every future request, client sends:
   Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

### What's Inside a JWT Token?

A JWT has three parts separated by dots: `header.payload.signature`

The **payload** in our case contains:
```json
{
  "userId": "clx1abc2d0001",
  "role": "VIEWER",
  "iat": 1711929600,      // issued at (timestamp)
  "exp": 1712534400       // expires at (7 days later)
}
```

The **signature** proves the token hasn't been tampered with. It's created using the `JWT_SECRET` from your `.env` file. If someone changes the payload, the signature won't match and `jwt.verify()` will throw an error.

### How the Token Is Checked on Every Request

This happens in `auth.middleware.ts`:

```typescript
export function verifyToken(req, _res, next) {
  // 1. Get the Authorization header
  const authHeader = req.headers.authorization;
  // Expected format: "Bearer eyJhbGciOiJIUzI1NiIs..."

  // 2. If there's no header or it doesn't start with "Bearer ", reject
  if (!authHeader || !authHeader.startsWith("Bearer ")) {
    return next(new ApiError(401, "Authentication token is required"));
  }

  // 3. Extract the token (everything after "Bearer ")
  const token = authHeader.split(" ")[1];

  // 4. Verify the token using the same secret that signed it
  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    // decoded = { userId: "abc123", role: "VIEWER", iat: ..., exp: ... }

    // 5. Attach the user info to the request object
    req.user = decoded;
    // Now ANY code after this middleware can access req.user.userId and req.user.role

    next(); // Continue to the next middleware/controller
  } catch {
    next(new ApiError(401, "Invalid or expired token"));
  }
}
```

The key insight: after `verifyToken` runs, `req.user` exists on the request object. Every controller can use `req.user.userId` to know WHO made the request and `req.user.role` to know WHAT they're allowed to do.

### Password Hashing вҖ” Why We Don't Store Plain Passwords

When a user registers, we don't store `"secret123"` in the database. We store a **hash** вҖ” a one-way scrambled version:

```typescript
const hashedPassword = await bcrypt.hash("secret123", 10);
// Result: "$2a$10$X7UrE5C..." (60 characters, looks like gibberish)
```

The `10` is the "salt rounds" вҖ” how many times to scramble it. Higher = more secure but slower.

When they log in, we compare the plain password to the hash:
```typescript
const isValid = await bcrypt.compare("secret123", storedHash);
// Returns true or false
```

This is one-way. Even if someone steals the database, they can't reverse the hash back to "secret123".

## 6. Roles вҖ” How VIEWER, ANALYST, and ADMIN Are Separated

### Where Roles Are Defined

In `prisma/schema.prisma`:
```prisma
enum Role {
  VIEWER
  ANALYST
  ADMIN
}
```

This creates a strict set of allowed values. A user can ONLY be one of these three. The database will reject anything else.

### How a User Gets a Role

When registering, you can optionally specify a role:
```json
{ "name": "Alice", "email": "...", "password": "...", "role": "ANALYST" }
```

If you don't specify one, you get `VIEWER` by default:
```typescript
role: input.role ?? "VIEWER"
```

### How Roles Are Enforced вҖ” The Two Layers

**Layer 1: At the app level (in `app.ts`)**

Some entire route groups are locked down:
```typescript
app.use("/api/users", verifyToken, requireRole("ADMIN"), userRoutes);
```
This means EVERY route under `/api/users/*` requires you to be an ADMIN. A VIEWER or ANALYST will get a 403 error before their request even reaches the user controller.

**Layer 2: At the individual route level (in `*.routes.ts`)**

Within a module, specific routes can have extra role requirements:
```typescript
// transaction.routes.ts
router.post("/",   requireRole("ADMIN"), transactionController.create);   // ADMIN only
router.get("/",    transactionController.getAll);                          // Anyone logged in
router.get("/:id", transactionController.getById);                        // Anyone logged in
router.patch("/:id", requireRole("ADMIN"), transactionController.update); // ADMIN only
router.delete("/:id", requireRole("ADMIN"), transactionController.remove); // ADMIN only
```

And for the dashboard:
```typescript
// dashboard.routes.ts
router.use(requireRole("ANALYST", "ADMIN"));  // Applied to ALL routes in this file
```

### The requireRole Function вҖ” How It Actually Works

```typescript
export function requireRole(...roles: Role[]) {
  // This is a "factory function" вҖ” it returns a middleware function
  return (req, _res, next) => {
    // At this point, verifyToken already ran, so req.user exists

    if (!req.user) {
      return next(new ApiError(401, "Authentication required"));
    }

    // Check: is the user's role in the list of allowed roles?
    if (!roles.includes(req.user.role)) {
      return next(new ApiError(403, "Insufficient permissions"));
    }

    next(); // Role is allowed, continue
  };
}
```

When you write `requireRole("ANALYST", "ADMIN")`, it creates a middleware that only allows requests where `req.user.role` is either `"ANALYST"` or `"ADMIN"`. If a VIEWER tries, they get a 403.

### The Full Permission Matrix

Here's what each role can access:

| Action                     | VIEWER | ANALYST | ADMIN |
|----------------------------|--------|---------|-------|
| Register / Login           |   Yes  |   Yes   |  Yes  |
| GET transactions           |   Yes  |   Yes   |  Yes  |
| POST/PATCH/DELETE transactions |  No |    No   |  Yes  |
| GET dashboard analytics    |   No   |   Yes   |  Yes  |
| GET/PATCH/DELETE users     |   No   |    No   |  Yes  |

## 7. Middlewares вҖ” The Security Guards of Every Route

Middlewares are functions that sit between the incoming request and your actual route handler. They run **in order**, one after another, like security checkpoints.

### The Three Middlewares in This Project

**1. `verifyToken` (auth.middleware.ts)**

- **Job**: Check if the request has a valid JWT token
- **If it passes**: Attaches `req.user = { userId, role }` and calls `next()`
- **If it fails**: Calls `next(new ApiError(401, ...))` вҖ” request dies here

**2. `requireRole(...roles)` (role.middleware.ts)**

- **Job**: Check if `req.user.role` is in the allowed list
- **Must run AFTER `verifyToken`** because it needs `req.user` to exist
- **If it passes**: Calls `next()`
- **If it fails**: Calls `next(new ApiError(403, "Insufficient permissions"))` вҖ” request dies here

**3. `errorHandler` (error.middleware.ts)**

- **Job**: Catch ANY error from anywhere and send a clean JSON response
- **This is special** вҖ” it has 4 parameters `(err, req, res, next)` instead of 3. Express uses this signature to identify error-handling middleware.
- **It runs LAST** вҖ” it's mounted at the bottom of `app.ts`

### How Errors Reach the Error Handler

In Express, when you call `next(someError)` from any middleware or controller, Express **skips all remaining normal middleware** and jumps straight to the error handler.

```
verifyToken вҶ’ requireRole вҶ’ controller вҶ’ service
     в”Ӯ              в”Ӯ            в”Ӯ           в”Ӯ
     в”Ӯ next(err)    в”Ӯ next(err)  в”Ӯ next(err) в”Ӯ throws
     в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                         в”Ӯ
                         в–ј
                    errorHandler
                         в”Ӯ
                         в–ј
                 { success: false, message: "..." }
```

### What the Error Handler Does

```typescript
function errorHandler(err, _req, res, _next) {
  // Is it our custom ApiError? (we threw it on purpose)
  if (err instanceof ApiError) {
    return res.status(err.statusCode).json({ success: false, message: err.message });
  }

  // Is it a Prisma unique constraint error? (e.g., duplicate email)
  if (err.code === "P2002") {
    return res.status(409).json({ success: false, message: "Already exists" });
  }

  // Is it a Prisma "not found" error?
  if (err.code === "P2025") {
    return res.status(404).json({ success: false, message: "Record not found" });
  }

  // Something we didn't expect вҖ” log it and send generic 500
  console.error("Unhandled error:", err);
  return res.status(500).json({ success: false, message: "Internal server error" });
}
```

This is why the client always gets a clean JSON error вҖ” never a raw stack trace or HTML error page.

## 8. Controller vs Service вҖ” Why We Split Them

### The Rule

> **Controllers** handle HTTP stuff (request, response, validation).
> **Services** handle business logic (database queries, calculations).

A service should NEVER know about `req`, `res`, or HTTP status codes. A controller should NEVER write Prisma queries.

### Why?

Because tomorrow, you might want to use the same `transactionService.create()` logic from:
- A REST API endpoint (controller)
- A GraphQL resolver
- A cron job that auto-creates transactions
- A CLI script

If the logic is in the service (which just takes plain data and returns plain data), you can reuse it everywhere. If it's mixed into the controller (which depends on `req` and `res`), you're stuck.

### How the Split Looks

**Controller** (HTTP layer):
```typescript
export async function create(req, res, next) {
  try {
    // 1. Validate the request body with Zod
    const validation = createTransactionSchema.safeParse(req.body);
    if (!validation.success) {
      res.status(400).json({ success: false, errors: validation.error.issues });
      return;
    }

    // 2. Call the service with clean, validated data
    const transaction = await transactionService.create(
      validation.data,    // just the data, no req/res
      req.user.userId     // who created it
    );

    // 3. Send the response
    ApiResponse(res, 201, "Transaction created successfully", transaction);
  } catch (error) {
    next(error); // Let error handler deal with it
  }
}
```

**Service** (business logic layer):
```typescript
export async function create(input: CreateTransactionInput, userId: string) {
  // Pure database operation. No req, no res, no HTTP knowledge.
  const transaction = await prisma.transaction.create({
    data: {
      amount: input.amount,
      type: input.type,
      category: input.category,
      date: new Date(input.date),
      notes: input.notes,
      createdById: userId,
    },
    include: {
      createdBy: {
        select: { id: true, name: true, email: true },
      },
    },
  });

  return transaction; // Just returns data. Controller decides what HTTP status to use.
}
```

### The try/catch Pattern

Every controller function wraps its body in `try/catch`:

```typescript
try {
  // ... validate, call service, send response
} catch (error) {
  next(error); // Forward to error handler
}
```

This is essential because async functions can throw errors (database down, validation fails in service, etc.). Without `try/catch`, the error would crash the server or send an ugly response. With it, the error gets forwarded to our global `errorHandler`.

## 9. Validation with Zod вҖ” Rejecting Bad Data Early

### The Problem

Users can send literally anything in a request body:
```json
{ "email": "not-an-email", "password": 42, "role": "SUPERADMIN" }
```

If we pass this to the database, bad things happen. We need to check the data BEFORE doing anything with it.

### How Zod Works

A Zod schema is a description of what valid data looks like:

```typescript
const registerSchema = z.object({
  name: z.string().min(1, "Name is required").max(100),
  email: z.string().email("Invalid email address"),
  password: z.string().min(6, "Password must be at least 6 characters").max(128),
  role: z.enum(["VIEWER", "ANALYST", "ADMIN"]).optional(),
});
```

This says:
- `name` must be a string, between 1 and 100 characters
- `email` must be a valid email format
- `password` must be a string, 6-128 characters
- `role` is optional, but if provided, must be one of the three enum values

### How We Validate

```typescript
const validation = registerSchema.safeParse(req.body);

if (!validation.success) {
  // Data is invalid вҖ” send back what's wrong
  res.status(400).json({
    success: false,
    errors: validation.error.issues
  });
  return; // Stop here. Don't call the service.
}

// Data is valid and typed! validation.data is now strongly typed.
const result = await authService.register(validation.data);
```

`safeParse` doesn't throw вҖ” it returns either:
- `{ success: true, data: { ... } }` вҖ” the validated, correctly-typed data
- `{ success: false, error: { issues: [...] } }` вҖ” an array of what went wrong

### Example Error Response

If someone sends `{ "email": "bad", "password": "12" }`:

```json
{
  "success": false,
  "errors": [
    { "code": "too_small", "path": ["name"], "message": "Required" },
    { "code": "invalid_string", "path": ["email"], "message": "Invalid email address" },
    { "code": "too_small", "path": ["password"], "message": "Password must be at least 6 characters" }
  ]
}
```

### All Zod Schemas in This Project

| Schema | File | Fields |
|--------|------|--------|
| `registerSchema` | auth.controller.ts | name, email, password, role? |
| `loginSchema` | auth.controller.ts | email, password |
| `updateRoleSchema` | user.controller.ts | role |
| `updateStatusSchema` | user.controller.ts | isActive |
| `createTransactionSchema` | transaction.controller.ts | amount, type, category, date, notes? |
| `updateTransactionSchema` | transaction.controller.ts | all fields optional (partial update) |

## 10. Prisma and the Database вҖ” How Data Is Stored and Retrieved

### What Is Prisma?

Prisma is an **ORM** (Object-Relational Mapper). Instead of writing raw SQL like:
```sql
SELECT * FROM users WHERE email = 'alice@example.com';
```

You write TypeScript:
```typescript
const user = await prisma.user.findUnique({ where: { email: "alice@example.com" } });
```

Prisma converts this to SQL, sends it to the database, and returns the result as a typed JavaScript object.

### The Schema вҖ” Your Database Blueprint

`prisma/schema.prisma` defines two tables:

**Users table:**
```prisma
model User {
  id           String        @id @default(cuid())    // Auto-generated unique ID
  name         String
  email        String        @unique                  // No two users can have same email
  password     String                                 // Stored as bcrypt hash
  role         Role          @default(VIEWER)         // Default role
  isActive     Boolean       @default(true)           // Can be deactivated by admin
  createdAt    DateTime      @default(now())          // Auto-set on creation
  updatedAt    DateTime      @updatedAt               // Auto-updated on every change
  transactions Transaction[]                          // One user вҶ’ many transactions

  @@map("users")                                      // Actual table name in PostgreSQL
}
```

**Transactions table:**
```prisma
model Transaction {
  id          String          @id @default(cuid())
  amount      Float                                    // Dollar amount
  type        TransactionType                          // INCOME or EXPENSE
  category    String                                   // "Salary", "Rent", "Food", etc.
  date        DateTime                                 // When the transaction happened
  notes       String?                                  // Optional notes (? means nullable)
  isDeleted   Boolean         @default(false)          // Soft delete flag
  createdAt   DateTime        @default(now())
  updatedAt   DateTime        @updatedAt
  createdBy   User            @relation(...)           // Who created this transaction
  createdById String                                   // Foreign key to users table

  @@map("transactions")
}
```

### Key Prisma Operations Used in This Project

**Create a record:**
```typescript
await prisma.user.create({
  data: { name: "Alice", email: "alice@example.com", ... }
});
```

**Find one record:**
```typescript
await prisma.user.findUnique({ where: { email: "alice@example.com" } });
```

**Find many with filtering and pagination:**


**Update a record:**
```typescript
await prisma.user.update({
  where: { id: "abc123" },
  data: { role: "ADMIN" },
});
```

**Delete a record (hard):**
```typescript
await prisma.user.delete({ where: { id: "abc123" } });
```

**Aggregate (sum, count):**
```typescript
await prisma.transaction.aggregate({
  where: { isDeleted: false, type: "INCOME" },

// Result: { _sum: { amount: 15000 } }
```

**Group by:**
```typescript

  by: ["category"],
  _sum: { amount: true },
  _count: { id: true },

// Result: [{ category: "Food", _sum: { amount: 500 }, _count: { id: 12 } }, ...]
```

**Raw SQL (for complex queries):**
```typescript

  SELECT EXTRACT(MONTH FROM "date")::int AS month, ...

`;
```

### The `select` vs `include` Difference

**`select`** вҖ” Pick specific fields. Other fields are excluded.
```typescript

});
```

**`include`** вҖ” Include related records (joins).
```typescript
prisma.transaction.findMany({
  include: {
    createdBy: {                          // Include the user who created it

    }
  }
});
```

## 11. Error Handling вҖ” What Happens When Things Go Wrong

### The Two Types of Errors

**1. Expected errors** вҖ” Things we know can go wrong
```typescript
if (!user) {
  throw new ApiError(404, "User not found");
}

if (!user.isActive) {
  throw new ApiError(403, "Account is deactivated");
}
```

These use our custom `ApiError` class. We choose the status code and message.

**2. Unexpected errors** вҖ” Things Prisma or other libraries throw
```
Prisma P2002: Unique constraint violation (duplicate email)
Prisma P2025: Record not found during update/delete
Database connection failed
Some random bug we didn't anticipate
```

### How They're All Caught

**In controllers** вҖ” `try/catch` catches everything and forwards it:
```typescript
try {
  // ... controller logic
} catch (error) {
  next(error); // Sends to errorHandler
}
```

**In the error handler** вҖ” different error types get different responses:

```
ApiError(404, "User not found")     вҶ’ 404 { success: false, message: "User not found" }
Prisma P2002 (duplicate)            вҶ’ 409 { success: false, message: "Already exists" }
Prisma P2025 (not found)            вҶ’ 404 { success: false, message: "Record not found" }
Anything else                       вҶ’ 500 { success: false, message: "Internal server error" }
```

### The ApiResponse Helper

For success responses, we use the `ApiResponse` helper for consistency:

```typescript
function ApiResponse(res, statusCode, message, data?) {
  res.status(statusCode).json({
    success: statusCode >= 200 && statusCode < 300,  // true for 2xx
    message,
    ...(data !== undefined && { data }),               // Include data only if provided
  });
}
```

This means every success response looks like:
```json
{
  "success": true,
  "message": "Transaction created successfully",
  "data": { ... }
}
```

And every error response looks like:
```json
{
  "success": false,
  "message": "User not found"
}
```

Consistent response shapes make life much easier for the frontend developer.

## 15. Soft Delete vs Hard Delete

### What's the Difference?

**Hard delete** = The row is permanently removed from the database. Gone forever.
```typescript
await prisma.user.delete({ where: { id } });
```

**Soft delete** = The row stays in the database, but a flag (`isDeleted`) is set to `true`. It's still there, but we pretend it doesn't exist.
```typescript
await prisma.transaction.update({
  where: { id },
  data: { isDeleted: true },
});
```

### Why Use Soft Delete for Transactions?

- **Audit trail** вҖ” You might need to see deleted transactions later for accounting purposes
- **Recovery** вҖ” An admin can "undelete" by setting `isDeleted` back to `false`
- **Data integrity** вҖ” If reports reference a transaction, hard deleting it breaks the reports

### How Soft Delete Is Enforced

Every single query that reads transactions includes `isDeleted: false`:

```typescript
const NOT_DELETED = { isDeleted: false } as const;

// In getAll:
const where = { ...NOT_DELETED, ... };

// In getById:
prisma.transaction.findFirst({ where: { id, ...NOT_DELETED } });

// In dashboard:
prisma.transaction.aggregate({ where: { ...NOT_DELETED, type: "INCOME" } });
```

The `NOT_DELETED` constant is defined once and spread into every query. This prevents the mistake of forgetting to filter deleted records in one place.

### What Uses What?

| Entity       | Delete Type | Why |
|--------------|-------------|-----|
| Transactions | Soft delete | Need audit trail, financial data should never truly vanish |
| Users        | Hard delete | Admin deliberately removing a user; cascades delete their transactions too |

## 16. The Singleton Pattern вҖ” Why Prisma Client Is Created Only Once

### The Problem

If every file that needs the database creates its own `new PrismaClient()`, you'll have dozens of connections to the database. Databases have connection limits (often 20-100). You'll run out fast.

### The Solution

Create ONE PrismaClient and share it everywhere:

```typescript
// db.ts
export const prisma = globalForPrisma.prisma ?? createClient();
```

Every file imports this same instance:
```typescript
import { prisma } from "../../config/db.js";
```

No matter how many files import it, they all get the exact same object.

### The globalThis Trick

```typescript
const globalForPrisma = globalThis as unknown as { prisma: PrismaClient | undefined };

// ...

if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma;
}
```

During development, tools like `tsx watch` restart your code when files change. Each restart re-imports all modules, which would normally create a new PrismaClient. By storing it on `globalThis` (which survives restarts), we reuse the same client.

In production, this isn't needed because the server starts once and stays running.

## 17. TypeScript, ESM, and How the Build Works

### Why TypeScript?

TypeScript adds **types** to JavaScript. Instead of discovering at runtime that `user.emial` is a typo (undefined), TypeScript catches it at compile time. In this project, Prisma generates types from the database schema, so `prisma.user.findUnique()` knows exactly what fields exist.

### What Is ESM?

ESM (ECMAScript Modules) is the modern JavaScript module system. It uses `import`/`export` instead of the old `require()`/`module.exports`.

In `package.json`:
```json
{ "type": "module" }
```

This tells Node.js: "All `.js` files in this project use `import`/`export`."

### Why Do Imports End in `.js`?

```typescript
import { prisma } from "../../config/db.js";  // .js, not .ts!
```

This is an ESM rule. When TypeScript compiles `db.ts` вҶ’ `db.js`, the import path must already point to the final `.js` file. TypeScript is smart enough to understand that `db.js` means `db.ts` during development.

### Development vs Production

**Development** (`npm run dev`):
```
tsx watch src/server.ts
```
`tsx` runs TypeScript directly вҖ” no compilation step. It watches for file changes and auto-restarts.

**Production** (`npm run build` then `npm start`):
```
1. tsc                    вҶ’ Compiles all .ts files to .js in /dist
2. node dist/server.js    вҶ’ Runs the compiled JavaScript
```

### tsconfig.json Highlights

```json
{
  "module": "nodenext",           // Output ESM-compatible JavaScript
  "moduleResolution": "nodenext", // Use Node.js ESM resolution rules
  "strict": true,                 // Enable all strict type checks
  "outDir": "./dist",             // Compiled JS goes here
  "rootDir": "./src",             // Source TS lives here
}
```

## 19. Common Mistakes and Things to Watch Out For

### 1. Forgetting `isDeleted: false` in a Transaction Query

If you write a new query for transactions and forget the soft delete filter, you'll return deleted records. Always spread `NOT_DELETED`:
```typescript
const where = { ...NOT_DELETED, ...otherFilters };
```

### 2. Returning the Password Hash

Never `select` the password field in Prisma queries:
```typescript
// BAD вҖ” password hash is sent to the client
await prisma.user.findUnique({ where: { id } });

// GOOD вҖ” explicitly select only safe fields
await prisma.user.findUnique({
  where: { id },
  select: { id: true, name: true, email: true, role: true },
});
```

### 3. Business Logic in Controllers

If you find yourself writing Prisma queries or complex calculations in a controller, stop. Move it to the service. The controller should only:
1. Validate input
2. Call a service function
3. Send the response

### 4. Missing `next(error)` in Catch Blocks

If you `catch` an error and don't call `next(error)`, the request will hang forever (no response is sent). Always forward errors:
```typescript
catch (error) {
  next(error); // NEVER forget this
}
```

### 5. Middleware Order Matters

`verifyToken` MUST run before `requireRole`. If you reverse them, `requireRole` tries to check `req.user.role` but `req.user` doesn't exist yet.

```typescript
// CORRECT
app.use("/api/users", verifyToken, requireRole("ADMIN"), userRoutes);

// WRONG вҖ” requireRole runs before verifyToken
app.use("/api/users", requireRole("ADMIN"), verifyToken, userRoutes);
```

### 6. The Error Handler Must Be Last

```typescript
// CORRECT вҖ” error handler after all routes
app.use("/api/auth", authRoutes);
app.use("/api/users", ...);
app.use(errorHandler);  // Last

// WRONG вҖ” routes registered after error handler won't be caught
app.use(errorHandler);
app.use("/api/auth", authRoutes);  // Errors here won't reach errorHandler
```

### 7. Not Running `prisma generate` After Schema Changes

If you change `prisma/schema.prisma`, you must run:
```bash
npm run prisma:generate    # Regenerates TypeScript types
npm run prisma:migrate     # Creates/updates database tables
```

Without this, your TypeScript code will use stale types and the database won't have the new columns.

|------|-----------------|

| `utils/ApiResponse.ts` | Helper to send consistent `{ success, message, data }` responses |
| `modules/auth/auth.routes.ts` | POST `/register`, POST `/login` |

| `modules/users/user.routes.ts` | GET, PATCH, DELETE user routes |
| `modules/users/user.controller.ts` | Validates role/status updates, calls user service |
| `modules/users/user.service.ts` | User CRUD queries with Prisma |

| `modules/dashboard/dashboard.service.ts` | Aggregation queries (sum, count, groupBy, raw SQL) |
| `prisma/schema.prisma` | Database table definitions (User, Transaction) |
| `prisma.config.ts` | Tells Prisma CLI where the database is |
